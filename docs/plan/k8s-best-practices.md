# Kubernetes Best-Practices Audit

Findings from a project-wide Kubernetes best-practices audit performed 2026-05-30. Items are grouped by theme; `docs/STATUS.md` Queue rows reference the anchors below.

**Severity legend:** ЁЯФ┤ high ┬╖ ЁЯЯб medium ┬╖ ЁЯЯв low.

**Verified-good (not flagged here):** GMC/AGC/proxy manager pods all set `runAsNonRoot` + RO rootfs + drop ALL + seccomp `RuntimeDefault`; NetworkPolicies are well-split (workload / proxy / AGC / k8s-API with documented kube-proxy DNAT notes); Secret cache `DisableFor` discipline + `WatchesMetadata` for credentials; finalizers + owner references for GC; HTTP/2 disabled on metrics/webhook (Rapid Reset CVE); admission webhook rejects privileged containers and cross-namespace SecretRef.

| B1 | тЬЕ | ~~Manager ClusterRole grants `rbac.authorization.k8s.io: escalate` тАФ lets a compromised GMC create Roles at any privilege level.~~ Fixed 2026-06-01 by removing per-tenant Role creation entirely. A compromised GMC cannot create arbitrary Roles, cannot escalate, and cannot bind its SA into any ClusterRole other than `agc-tenant-role`. `roles: {get,delete}` retained for legacy GC of pre-fix per-tenant Roles. | [actionsgateway_controller.go:10](../../cmd/gmc/internal/controller/actionsgateway_controller.go), [builder.go (`buildAGCRoleBinding`)](../../cmd/gmc/internal/controller/builder.go), [agc_tenant_role.yaml](../../cmd/gmc/config/agc-tenant-role/agc_tenant_role.yaml) | тАФ |
| B2 | тЬЕ | ~~`patch`/`update` on `namespaces` cluster-wide so the GMC can stamp PSA labels тАФ also lets a compromised pod relabel `kube-system` PSA to `privileged`.~~ Fixed (Q56) via the `namespace-psa-guard` ValidatingAdmissionPolicy ([namespace-psa-guard.yaml](../../cmd/gmc/config/admission-policy/namespace-psa-guard.yaml), applied by `make deploy`). Scoped to the GMC ServiceAccount, it denies any namespace `UPDATE` unless the existing namespace carries the trusted marker label `actions-gateway.github.com/tenant: "true"` (read from `oldObject`, unforgeable), and denies changes to any label other than the six `pod-security.kubernetes.io/*` keys or to annotations. RBAC stays cluster-wide (RBAC cannot express a label selector); the VAP is the gate. A blanket privileged-value ban was rejected because `securityProfile: privileged` is a supported per-tenant opt-in тАФ the marker scope confines the blast radius to managed tenants. See [q56-namespace-patch-vap.md](q56-namespace-patch-vap.md). | [namespace-psa-guard.yaml](../../cmd/gmc/config/admission-policy/namespace-psa-guard.yaml), [actionsgateway_controller.go](../../cmd/gmc/internal/controller/actionsgateway_controller.go) (`applyNamespacePSA`) | тАФ |
| B3 | тЬЕ | ~~`applyNamespacePSA` silently overwrites admin edits тАФ no resourceVersion precondition, no event when admin intent loses.~~ Converted 2026-06-01 to Server-Side Apply with field manager `actionsgateway-controller-psa`; a 409 conflict on the first apply emits a `PSALabelsOverridden` Warning Event on the ActionsGateway before retrying with `ForceOwnership`. | [actionsgateway_controller.go](../../cmd/gmc/internal/controller/actionsgateway_controller.go) (`applyNamespacePSA`) | тАФ |

The RBAC `list` verb is retained (RBAC has no glob `resourceNames` and agent Secret names are dynamic); the application-level bulk-body exposure is closed and the residual RBAC `list` capability is the accepted [H-2 residual](security.md#h-2-agc-namespaced-role-grants-broad-secret-readlistwatch--createdelete). Covered by `TestPool_EnumeratesSecretsAsMetadataOnly`. | [pool.go (`listSecretMeta`/`getSecret`)](../../cmd/agc/internal/agentpool/pool.go) | тАФ |

## C. Worker / proxy pod security defaults тЬЕ

The operator-created pods (worker pods from AGC, proxy pods from GMC) are the highest-blast-radius surface. PSA `baseline` (the project default) does **not** enforce `runAsNonRoot` or seccomp. Proxy pods (GMC `buildProxyDeployment`) already ship a full hardened `SecurityContext` + resources, so the findings below cover **worker pods only**; all are fixed.

| # | Sev | Finding | Location | Fix |
|---|---|---|---|---|
| C1 | тЬЕ | ~~Worker pods get no default `SecurityContext` stamped тАФ `runAsNonRoot`, RO rootfs, capability drop, seccomp absent unless tenant `PodTemplate` provides them.~~ Fixed: `buildPod` now gap-fills a **profile-aware** secure-by-default SecurityContext via `applySecurityDefaults`. `baseline` (default) gets pod-level `runAsNonRoot` + seccomp `RuntimeDefault`; `restricted` adds the per-container `allowPrivilegeEscalation:false` + drop-ALL floor (required for that namespace to admit the pod); `privileged` opts out entirely. `SECURITY_PROFILE` is plumbed GMCтЖТAGC. **Deviation:** `readOnlyRootFilesystem` is deliberately *not* defaulted тАФ the runner writes to its work/home dirs and RO-rootfs is not part of the PSA-restricted floor; tenants can opt in. Documented in [05-security.md ┬з5.3](../design/05-security.md#53-security-profiles-and-the-privileged-opt-in). | [provisioner.go](../../cmd/agc/internal/provisioner/provisioner.go) (`applySecurityDefaults`) | тАФ |
| C2 | тЬЕ | ~~Worker pods have no default resource requests/limits when the tenant template omits them тЖТ Best-Effort, evictable, burns the eviction-retry budget fast.~~ Fixed: `applyResourceDefaults` stamps `500m`/`1Gi` requests+limits (Guaranteed QoS) on every profile when the tenant container declares neither. | [provisioner.go](../../cmd/agc/internal/provisioner/provisioner.go) (`applyResourceDefaults`) | тАФ |
