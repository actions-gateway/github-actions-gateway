Findings from a project-wide Kubernetes best-practices audit performed 2026-05-30. Items are grouped by theme; `docs/STATUS.md` Queue rows reference the anchors below.

**Severity legend:** 🔴 high · 🟡 medium · 🟢 low.

### A6 design note: `CreateOrPatch` vs Server-Side Apply

`CreateOrPatch` is **not** universally better than Server-Side Apply (SSA); it was the better fit for *this* migration. Both are strict improvements over the old `Get → full-object Update`, which carried the object's `resourceVersion` as a precondition (so a racing write produced a spurious `409 IsConflict` + requeue) and replaced the whole object (clobbering fields the controller doesn't manage). The choice between the two:

**Why `CreateOrPatch` here**

- **Typed builders vs. applyconfigurations.** The helpers already receive fully-built *typed* objects from the `build*` functions, so `CreateOrPatch` slots in with a small mutate closure. Doing SSA *correctly* needs applyconfiguration builders (`appsv1ac.Deployment(...)`): a typed Go struct can't distinguish "field unset" from "field == zero value", so applying a typed object claims ownership of every zero-valued field and fights other managers / strips defaults. Clean SSA would mean rewriting all ten builders into a parallel applyconfiguration set — far larger and riskier than the finding warranted.
- **Scope.** The finding (and [Q65](../STATUS.md)) explicitly specified `CreateOrPatch`.
- **Sufficiency.** The GMC reconciler is single-writer (`MaxConcurrentReconciles=1`), so `CreateOrPatch`'s no-precondition merge fully closes the conflict window; SSA's extra machinery buys little.

**Where SSA is the right tool (and why `applyNamespacePSA` keeps it)**

- **Ownership / contention detection.** SSA records per-field-manager ownership and surfaces a conflict when another manager (e.g. a human admin) changes a field you own. That is the *requirement* for `applyNamespacePSA` — PSA labels are a security control, so an out-of-band edit must be detected and reported (Warning Event), not silently re-asserted. See B2/B3.
- SSA also avoids the read round-trip, merges lists by schema key (not RFC-7386 whole-list replace), and produces genuinely empty diffs when nothing changed.

**The one wart of the `CreateOrPatch` approach.** Because each closure does `obj.Spec = desired.Spec` (omitting server-defaulted fields like a Deployment's rollout `Strategy` or an HPA's `Behavior`), it emits a *non-empty* patch every reconcile that the apiserver then no-op-dedups (re-applies defaults, skips the write, no `resourceVersion` bump). SSA-with-applyconfigurations would emit no patch traffic at all. So `CreateOrPatch` here deliberately relies on the apiserver's no-op-write detection — which is precisely what `apply_nochurn_test.go` exists to verify, since a fake client reproduces neither defaulting nor no-op dedup.
